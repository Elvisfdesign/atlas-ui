<DocsLayout>

<DocsTitle>AI Assistant State</DocsTitle>

<DocsIntro>
Consolidates "AI Processing / Error / Empty State" — named as one item
in the component brief — into a single component with a `status` prop,
the same pattern Feedback's Empty State uses for Empty/Success/Error.
</DocsIntro>

<DocsSection title="Overview">

An icon-chip + title + description block for Assistant Panel's message
area: shown while a response is generating, when a request fails, or
before the first message is sent.

</DocsSection>

<DocsSection title="Live Preview">

<Canvas of={AIAssistantStateStories.Processing} />

</DocsSection>

<DocsSection title="Usage">

Swap `status` between `processing`, `error`, and `empty` rather than
building three separate components — each gets its own icon and tone,
sharing one layout.

</DocsSection>

<DocsSection title="Variants">

<Canvas of={AIAssistantStateStories.ErrorState} />
<Canvas of={AIAssistantStateStories.Empty} />

</DocsSection>

<DocsSection title="Sizes">

One size — a centered block meant to fill Assistant Panel's message
area.

</DocsSection>

<DocsSection title="States">

`processing`, `error`, and `empty` are shown above; each accepts an
optional `action` (e.g. a "Try again" button for `error`).

</DocsSection>

<DocsSection title="Accessibility">

- `processing` carries `role="status"` so assistive tech is told work
  is ongoing, matching Feedback's Loading State.
- `error` and `empty` render as plain content — they're not
  interruptive announcements, consistent with Feedback's Empty State.
- The icon is always decorative (`aria-hidden`); the title and
  description carry the meaning.

</DocsSection>

<Controls of={AIAssistantStateStories.Processing} />

<ArgTypes of={AIAssistantStateStories} />

</DocsSection>

<DocsSection title="Examples">

```tsx
import { AIAssistantState } from '@/components/ai/AIAssistantState';

<AIAssistantState
  status="processing"
  title="Thinking…"
  description="Reading the document and extracting fields."
/>
```
